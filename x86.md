# x86.md — the seed for the first x86 build session

Paste this as the opening message of a fresh session on the x86-64 Kali desktop,
once the toolchain is installed (or `scripts/setup/x86_bootstrap.sh` has run). It
orients you to the repo, sets the working discipline, and gives the first concrete
step. This is the start of a **multi-session** build — you are not here to finish a
pipeline in one sitting; you are here to build a clean, long-term, tool-first
solution one empirically-verified step at a time, leaving the repo cleaner and
better-documented than you found it every session.

## Your job

Continue an astrophotography **checklist + knowledge workspace** on its target
hardware. Official industry tools do ALL pixel work AND all measurement; the repo's
own code only ORCHESTRATES, RECORDS, RESEARCHES, and fills gaps with sanctioned
standalone detectors. Bring the durable core (stack → solve → SPCC → compose) and
the tool-first render up to best-in-class on x86 — measured, reproducible, honest.

## Read before you act — the repo read-order (load ALL of these, then assess state)

   anti-drift test, the environment, the binding rules. This overrides defaults.

6. **docs/x86-empirical-test-plan.md** + **docs/x86-setup-and-install.md** — the
   ordered build/test plan and the install reference.

Do not start work until these are loaded and you can state the current state.

## The discipline — enforce it on yourself AND on the code

- **Best long-term professional solution, always. No bandaids, no workarounds.**
  Fix the ROOT cause, never the symptom. A step whose only purpose is to hide what a
  prior step broke is a bandaid — reject it and fix the cause.
- **Official tools do all pixel work and measurement.** In-house code may only
  orchestrate / record / research / gap-fill, and only if it carries a removal
  condition. If in-house code reads, transforms, analyzes, or gates the deliverable's
  pixels, or reimplements a measurement a tool already provides — **that is a BUG.**
  Treat every violation of the repo's rules in code as a bug, not a style nit.
- **Obvious bug → stop and fix it on the spot** (clear root cause, a quick correct
  fix, verified end-to-end). Don't leave it; don't route around it.
- **Long-term / architectural fix → stop and add it to BACKLOG** with its root cause.
  Do NOT half-implement it and do NOT bandaid around it — scope it honestly and come
  back to it as its own measured step.
- **A gap (no official tool provides the mechanism) → research it and document it
  honestly.** Never a silent numpy substitute. Call out every gap; a documented gap
  is honest, a hidden one is a lie. If it's major, land a cited `docs/` deep-dive.
- **Research is standing work.** Before relying on any doc claim, pause and confirm
  the current best tool/version from PRIMARY sources + the community. A program's
  problem almost certainly has a forum answer — find it; you are not the first to hit it.
- **Read and maximize what's already installed — FIRST.** Read the tool's own
  `--help` and docs and confirm its capability EMPIRICALLY on the real binary before
  adding a tool or writing a line of code. This repo's whole history is discovering the
  tool already did what we were about to hand-roll.
- **Nothing is final until empirically tested on real data.** A doc reading or
  mechanism analysis is a HYPOTHESIS — mark it as such, name the test, run it, graduate
  the result. This is your rig; the "verify on x86" flags in the docs are now yours to close.
- **This is not about finishing in one session.** A clean, correct, documented
  increment beats a fast, complete, fragile one. Leave provisional things marked provisional.

## Why the strict industry protocol matters — the north star, never optimize against it

The process this repo drives is complex, ever-changing, and freely improving, and
**AI cannot natively judge an astro image.** Strict adherence to standard industry
tools and workflow is exactly what makes it tractable:

- **Researchable** — a standard tool's problem has a community and a posted answer; a
  hand-rolled reinvention has neither. Standard protocol means help exists.
- **Comparable** — a standard control (a pinned, reproducible chain + a reference
  dataset suite) lets results be compared honestly across every change.
- **Clean** — official tools + thin orchestration keep the environment maintainable,
  not a drift of one-off scripts.

Every decision serves these three. A shortcut that trades any of them away is the
wrong decision, however fast.

## Step 1 — verify + inventory the toolchain (before ANY build work)

"Installed" is not "works headless at the pinned version, producing correct output."
Verify empirically and reconcile against the manifest:

1. For each tool (Siril, **official** GraXpert — not the `geeksville` fork, RC-Astro
   `rc-astro` v1.0.0, StarNet2, DeepSNR, Cosmic Clarity, ASTAP, astropy, darktable-cli,
   `lensfun-update-data`, hugin-tools `cpfind`/`cpclean`/`autooptimiser`): confirm it
   runs headless at the expected version and produces expected output on a throwaway
   synthetic frame. Then run the per-rig undistort setup trio — `lensfun-update-data`,
   `scripts/darktable/install_styles.sh`, `scripts/darktable/install_lens_model.sh`
   (the fitted entry + the vignetting/tca strip; re-run after every DB update —
   and verify with the uniform-card check: warp a uniform gray card through
   `lensdist`, corner medians must equal centre via Siril `stat`; darktable
   ignores a style's lens op_params, so only the stripped DB keeps vignetting
   out — `docs/dead-ends.md`) — and prove it end-to-end with
   `scripts/stack/lens_preflight.py <session> <set> --require-profile` on the first
   staged set (darktable never fails loudly on a lens mismatch).
2. **Capture the tool surfaces the research left open on x86** (research is standing
   work): run each tool's no-arg / `--help` and record its REAL CLI flags — above all
   `rc-astro nxt` (the exact chroma flag spelling, e.g. `denoise_color`, that closes
   the chroma-noise gap), `rc-astro bxt --correct-only`, StarNet2 `-n/--unscreen` and
   `-d`, and official GraXpert's flags. Update TOOLS.md / the deep-dives to what is
   actually there; file any surprise as a gap or a BACKLOG item.
3. Confirm the upgrades the research assumed: **astropy present** (unblocks the
   FITS-I/O retirement), **official GraXpert** (not the fork), the Siril version, and
   the no-GPU CPU path (`--benchmark-all` per neural tool to pin the fastest device;
   self-measure wall-clock — no vendor numbers exist).

Anything that fails or differs: obvious fix → fix it; needs design → BACKLOG it;
either way, record what is real.

## Step 2 — re-establish the durable core and pin it as the standard control

Bring the durable core (stack → solve → SPCC → compose) up on x86 and confirm it
runs reproducibly — for the wide-field UNTRACKED class the stack builder is
`scripts/stack/run_undistort_pipeline.sh` (it consumes each set's ratified
`recipe.json` cull automatically). All five july14 sets are PREPPED and the
5-set combine is RENDERED + APPROVED on the arm rig (tag
`july14-all5-cov25frame-approved`); the queued x86 stack jobs are therefore
RE-MEASURES, in order: **the 5-set re-compose in ONE registration** (all 107
kept sub-stacks — the arm rig's two-interleaved-half adaptation retires on
1 TB disk; the removal register row names it), which also pays the
**item-7 groups-vs-single-pass validation debt** the approved render rides
on, then **set-01 single-pass at full depth** (369 post-cull frames — the
arm disk ceiling forced the groups route; single-pass needs the raws
re-staged). Solve → SPCC K factors compare against the tracked records
(`qa_work/spcc_set-01.json`; per-set flats per the ratified rule — set
raws re-stage from off-rig as needed). Compare only against
post-vignetting-strip records (`qa_work/gradient_qa.json`); a stack
predating the strip carries the corner bowl and is not a regression
baseline. That pinned chain (tool versions / params / seeds) + the
reference dataset suite IS the **standard control** every later change is measured
against (README "How a change is accepted": reproducibility + no-regression +
declared delta). Baseline it. Do not tune any look before the control exists — it
is what keeps results comparable and the process honest.

## Step 3+ — the phased build, one measured step at a time

Follow docs/x86-empirical-test-plan.md (Phase 0→5). Each step is ONE bracketed change
with a pre-registered hypothesis, driven by an official tool, tested on real data /
full-frame lossless finals, closed with a **WIN | NULL | needs-eyes** verdict into the
tracked per-dataset ledger. A killed hypothesis becomes a dead-end entry in
docs/dead-ends.md WITH ITS NUMBERS before anything else is tried. **Phase 4 (the
render toolkit) builds ON the arm-rig render-tier results** — whatever ladders
`docs/render-tier-arm-plan.md` closed on the base rig arrive here as measured
priors to re-measure, and the x86-unlocked tiers (StarNet2/SXT separation,
BXT/NXT, Cosmic Clarity, DeepSNR) layer on top. The remaining cleared
retirements — the last astropy FITS-I/O site + the `compose.py` combine →
`rgbcomp` (both data-gated on a multi-channel set; BACKLOG item 6), the 16-bit
stack-time intermediates (register row) — are natural early wins, but each is
done, verified, and committed as its own clean step, never batched.

## When you hit something — the decision tree

- **In-house code that breaks the tool-first rule** → it is a bug. Obvious replacement
  with a tool now → do it; needs design → BACKLOG with the root cause.
- **Obvious bug** → fix on the spot, verify, move on.
- **Long-term / architectural fix** → STOP, add to BACKLOG with the root cause; do not
  bandaid around it.
- **A gap (no tool does it)** → research primary sources + community, document it
  honestly (TOOLS.md, and a cited `docs/` deep-dive if major); never a silent hand-roll.
- **A provisional claim** ("likely / predicted / verify on x86") → test it on this rig,
  then graduate the result into the operating docs and mark it tested.

## Leaving the session clean

Stop when the repo is in a clean, honest state — not when the code is "done." That
means: every change is verified or explicitly marked provisional; new gaps and
long-term fixes are in BACKLOG; killed hypotheses are in docs/dead-ends.md with their
numbers; TOOLS.md reflects the real installed surfaces; nothing is half-implemented or
bandaged. The next session picks up from a true picture.
